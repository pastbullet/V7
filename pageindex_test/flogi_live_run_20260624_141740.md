下面按“定位/目的 → 帧与 Payload 结构 → 标准流程 → 分支与异常 → 相关机制（尤其 Virtual Fabrics）”系统梳理 FLOGI。

## 1. FLOGI 是什么、处在什么位置

FLOGI（F_Port Login）是 ELS（Extended Link Service，扩展链路服务）之一；在 ELS 命令码表中，FLOGI 的命令值是 `04h`。<cite doc="FC-LS.pdf" page="37"/>  
ELS 的通用规则是：请求和应答各自都是单个 Sequence，请求/应答在一个 Exchange 内完成，ELS 帧的 `TYPE` 必须为 `01h`。<cite doc="FC-LS.pdf" page="36"/>

从登录模型看，Login 过程用于让 Nx_Port 与 Fabric（若存在）以及其它目的 Nx_Port 建立运行环境；其中 Fabric Login 与 N_Port Login 过程相似，只是 `D_ID`，以及可能的 `S_ID` 不同。<cite doc="FC-LS.pdf" page="152"/>  
在有 Fabric 的情况下，**FLOGI 必须先于所有其它 ELS**。<cite doc="FC-LS.pdf" page="36"/>  
另外，所有 Nx_Port 都要求先完成与 Fabric 的登录，**在 Fabric Login 完成前，不应尝试与其它 Nx_Port 通信**。<cite doc="FC-LS.pdf" page="153"/>

## 2. FLOGI 的目的：它完成哪些事

Fabric Login 主要完成以下功能：

| 作用 | 说明 |
|---|---|
| 检测是否存在 Fabric | 确定链路对端是不是 Fabric。<cite doc="FC-LS.pdf" page="153"/> |
| 获取 Fabric 全局运行特性 | 若存在 Fabric，Nx_Port 获得整个 Fabric 的运行特征、`F_Port_Name`、`Fabric_Name`。<cite doc="FC-LS.pdf" page="153"/> |
| 向 Fabric 上报本端身份/能力 | 向 Fabric 提供 Nx_Port 的运行特性、`N_Port_Name`、`Node_Name`。<cite doc="FC-LS.pdf" page="153"/> |
| 分配或确认 `N_Port_ID` | Fabric 可为发起登录的 Nx_Port 分配，或确认其 `N_Port_ID`。<cite doc="FC-LS.pdf" page="153"/> |
| 初始化 BB Credit | 初始化 buffer-to-buffer Credit。<cite doc="FC-LS.pdf" page="153"/> |
| 使能后续认证 | 若双方支持认证，可进入后续 Nx_Port-Fabric Authentication。<cite doc="FC-LS.pdf" page="153"/> |
| 使能后续 VF 协商 | 若双方支持 Virtual Fabrics，可进入后续 VF 参数协商。<cite doc="FC-LS.pdf" page="153"/> |

所以，FLOGI 不只是“拿一个地址”，它是 **Fabric 环境发现 + 身份交换 + 基础链路参数协商 + 后续认证/VF 协商入口**。<cite doc="FC-LS.pdf" page="153"/>

## 3. FLOGI 的帧/Payload 结构

### 3.1 ELS 层面的通用帧规则

先看 ELS 共性：

- `TYPE = 01h`。<cite doc="FC-LS.pdf" page="36"/>
- `R_CTL` 使用 ELS 规定的类别：请求为 `Request`，应答为 `Reply`。<cite doc="FC-LS.pdf" page="36"/>
- 请求或应答的第一个帧中，Payload 第一个字节/第一字承载 `ELS_Command code`；后续帧若有，只携带补充 Payload，不重复命令码。<cite doc="FC-LS.pdf" page="36"/>
- 一个 ELS 请求与对应应答在一个 Exchange 内完成；`LS_ACC` 在最后一帧置 `Last Sequence bit=1` 来终止 Exchange。<cite doc="FC-LS.pdf" page="36"/>

### 3.2 FLOGI 的命令码

- FLOGI 的 `ELS_Command` 值是 `04h`。<cite doc="FC-LS.pdf" page="37"/>

### 3.3 FLOGI / PLOGI / LS_ACC 的统一 Payload 总布局

标准把 FLOGI、PLOGI 和 LS_ACC 的总体 Payload 布局统一定义为表 149。其布局可概括为：

| Word 范围 | 内容 |
|---|---|
| Word 0 | `ELS_Command code` |
| Word 1-4 | `Common Service Parameters`（16 bytes） |
| Word 5-6 | `Port_Name` |
| Word 7-8 | `Node_Name` 或 `Fabric_Name` |
| Word 9-12 | Class 1 / Class 6 Service Parameters（16 bytes） |
| Word 13-16 | Class 2 Service Parameters（16 bytes） |
| Word 17-20 | Class 3 Service Parameters（16 bytes） |
| Word 21-24 | Obsolete |
| Word 25-28 | Vendor Version Level（16 bytes） |
| Word 29-30 | Services Availability（8 bytes，可选） |
| Word 31 | Login Extension Data Length（可选） |
| Word 32-61 | Reserved（可选扩展区内） |
| Word 62-63 | Clock Synchronization QoS（8 bytes，可选） |
| Word 64-n | Login Extension Data（若有） |

以上来自表 149。并且当 `Payload Bit = 0` 时，这些扩展字段（如 Services Availability、Login Extension Data 等）**不出现**，此时 Payload 长度为 **116 bytes**。<cite doc="FC-LS.pdf" page="166"/>

### 3.4 FLOGI 请求中具体带什么

显式 Fabric Login 请求规定：

- FLOGI 在一个**新的 Exchange**中发送；<cite doc="FC-LS.pdf" page="153"/>
- Payload 包含：
  - Nx_Port 的 `Service Parameters`
  - 64-bit `N_Port_Name`
  - 64-bit `Node_Name`。<cite doc="FC-LS.pdf" page="153"/>

这与表 149 的统一布局相吻合：Word 1-4 是 Common Service Parameters，Word 5-6 是 Port_Name，Word 7-8 是 Node_Name/Fabric_Name。<cite doc="FC-LS.pdf" page="166"/>

### 3.5 FLOGI 请求的 Common Service Parameters（表 151）

表 151定义了 **FLOGI 请求**中 Common Service Parameters 的字布局：

| Word | 字段 |
|---|---|
| 0 | `FC-PH Version - obsolete` + `Buffer-to-buffer Credit` |
| 1 | `Common Features` + `BB_SC_N` + `Buffer-to-buffer Receive Data Field size` |
| 2 | `Reserved` + `Total Concurrent Sequences` + `Relative offset by Information Category` |
| 3 | `E_D_TOV` |

<cite doc="FC-LS.pdf" page="169"/><cite doc="FC-LS.pdf" page="170"/>

### 3.6 FLOGI LS_ACC 的 Common Service Parameters（表 153）

表 153定义了 **FLOGI 成功应答 LS_ACC**中的 Common Service Parameters：

| Word | 字段 |
|---|---|
| 0 | `FC-PH Version - obsolete` + `Buffer-to-buffer Credit (Fx_Port)` |
| 1 | `Common Features` + `BB_SC_N` + `Buffer-to-buffer Receive Data Field size` |
| 2 | `R_A_TOV` |
| 3 | `E_D_TOV` |

<cite doc="FC-LS.pdf" page="170"/>

这说明请求与应答的 Common 参数并不完全对称：  
FLOGI 请求的 Word 2 是 `Total Concurrent Sequences` / `Relative offset by Information Category`，而 FLOGI LS_ACC 的 Word 2 则变成 `R_A_TOV`。<cite doc="FC-LS.pdf" page="170"/>

### 3.7 FLOGI 请求中哪些 Common Features 有意义

表 150给出了 Common Service Parameters 在 PLOGI/FLOGI/FLOGI LS_ACC 中的适用性。对 FLOGI 来说，比较关键的是：

- `Buffer-to-buffer Credit`：FLOGI 中适用。<cite doc="FC-LS.pdf" page="167"/>
- `Virtual Fabrics bit`（Word 1, bit 30）：**FLOGI 和 FLOGI LS_ACC 中适用**。<cite doc="FC-LS.pdf" page="167"/>
- `Valid Vendor Version Level`（Word 1, bit 29）：FLOGI 中适用，但 FLOGI LS_ACC 中不适用。<cite doc="FC-LS.pdf" page="167"/>
- `Multiple N_Port_ID Support`：FLOGI 中适用，但 FLOGI LS_ACC 中不适用。<cite doc="FC-LS.pdf" page="167"/>
- `Multiple N_Port_ID Assignment`：FLOGI LS_ACC 中适用，但 FLOGI 请求中不适用。<cite doc="FC-LS.pdf" page="167"/>
- `N_Port/F_Port`（Word 1, bit 28）在 FLOGI/FLOGI LS_ACC 中适用；注释规定 `N_Port/F_Port=0` 表示 N_Port，`=1` 表示 F_Port。<cite doc="FC-LS.pdf" page="168"/>
- `Payload Bit`（Word 1, bit 16）在 FLOGI/FLOGI LS_ACC 中适用，用来决定扩展 Payload 是否存在。<cite doc="FC-LS.pdf" page="168"/><cite doc="FC-LS.pdf" page="166"/>

### 3.8 FLOGI 中 Class Service Parameters 结构

表 156给出 **FLOGI 请求**中的 Class Service Parameters（每个 class 一组 16 bytes）：

| Word | 字段 |
|---|---|
| 0 | `Service Options` + `Initiator Control` |
| 1 | `Recipient Control` + `Reserved` |
| 2 | `Reserved` + `Total Concurrent Sequences` + `Nx_Port End-to-end Credit` |
| 3 | `Reserved` + `Open Sequences per Exchange` + `Reserved` |

<cite doc="FC-LS.pdf" page="179"/>

表 158给出 **FLOGI LS_ACC**中的 Class Service Parameters：

| Word | 字段 |
|---|---|
| 0 | `Service Options` + `Reserved` |
| 1 | `Recipient Control` + `Reserved` |
| 2 | `Reserved` + `Reserved` |
| 3 | `CR_TOV` |

<cite doc="FC-LS.pdf" page="179"/>

并且 Class validity 位（word 0, bit 31）表示该 Class 是否被支持：`0` 表示无效/不支持，`1` 表示有效/支持。<cite doc="FC-LS.pdf" page="179"/>

## 4. FLOGI 的完整标准流程

## 4.1 总体流程骨架

显式登录的一般定义是：使用 Login ELS（FLOGI 或 PLOGI）在新的 Exchange 中传送发起方 Service Parameters，响应方用 `LS_ACC` 返回自己的 Service Parameters。<cite doc="FC-LS.pdf" page="152"/>

对 FLOGI 具体化后，完整流程可总结为：

1. Nx_Port 判断需要进行显式 Fabric Login；<cite doc="FC-LS.pdf" page="152"/><cite doc="FC-LS.pdf" page="153"/>
2. 在**新的 Exchange**中发送 FLOGI；<cite doc="FC-LS.pdf" page="153"/>
3. 设置目标地址为 Fabric 的知名 F_Port 地址 `FFFFFEh`；<cite doc="FC-LS.pdf" page="153"/>
4. 按本端状态设置 `S_ID`（000000h / 0000h||YY / 已知 XXXXXX）；<cite doc="FC-LS.pdf" page="153"/>
5. Payload 中带上 Service Parameters、`N_Port_Name`、`Node_Name`；<cite doc="FC-LS.pdf" page="153"/>
6. 接收返回分支：`LS_ACC` / `F_BSY` / `P_BSY` / `F_RJT` / `P_RJT`。<cite doc="FC-LS.pdf" page="154"/>
7. 若收到正常 Fabric `LS_ACC`，完成 Fabric 登录，之后可继续与其它 Nx_Port 操作或进入 N_Port Login；若涉及认证或 Virtual Fabrics，再进入后续流程。<cite doc="FC-LS.pdf" page="154"/><cite doc="FC-LS.pdf" page="153"/>

## 4.2 请求阶段细化

### 4.2.1 Exchange 与地址设置

显式 Fabric Login 请求要求：

- Nx_Port **必须在新的 Exchange 中发送 FLOGI**；<cite doc="FC-LS.pdf" page="153"/>
- Nx_Port 分配一个 `OX_ID`；<cite doc="FC-LS.pdf" page="153"/>
- `D_ID` 设置为知名 F_Port 地址 `FFFFFEh`。<cite doc="FC-LS.pdf" page="153"/>

### 4.2.2 `S_ID` 的三种用法

这是 FLOGI 很关键的实现点：

| 本端状态 / 需求 | FLOGI 中的 `S_ID` | 含义 |
|---|---|---|
| 本端未标识，由 Fabric 完整分配地址 | `000000h` | 请求 Fabric 分配全部 24 bit 的 `N_Port_ID`。<cite doc="FC-LS.pdf" page="153"/> |
| 本端希望保留低 8 bit（如 AL_PA） | `0000h || YY` | 请求 Fabric 分配高 16 bit（23:8），并校验低 8 bit（7:0）。<cite doc="FC-LS.pdf" page="153"/> |
| 本端已知/自选完整地址 | `XXXXXX` | 用该地址发起登录；后续 Fabric 可确认或拒绝。<cite doc="FC-LS.pdf" page="154"/> |

标准特别说明：`0000h || YY` 的示例场景是 FC-AL-2，其中 `N_Port_ID` 低 8 bit 是 `AL_PA`。<cite doc="FC-LS.pdf" page="153"/>

## 5. FLOGI 的响应分支与后续动作

标准在“Responses to Explicit Fabric Login”里列了完整分支。

### 5.1 正常 Fabric 响应：LS_ACC，且 N_Port/F_Port=1（Fx_Port）

这是**标准的正常 Fabric 登录成功**分支。其语义是：

- `OX_ID` 与原 FLOGI 相同；<cite doc="FC-LS.pdf" page="154"/>
- `Common Service` 里的 `N_Port/F_Port` 位为 `1`，表示对端是 `Fx_Port`；<cite doc="FC-LS.pdf" page="154"/><cite doc="FC-LS.pdf" page="168"/>
- `D_ID` 为 Fabric 分配给该 Nx_Port 的 `N_Port_ID`。规则如下：  
  - 若原 `S_ID=000000h`，则返回 `D_ID=XXXXXX`；  
  - 若原 `S_ID=0000h || YY`，则返回 `D_ID=XXXXYY`；  
  - 若原 `S_ID=XXXXXX`，则返回同样的 `XXXXXX`。<cite doc="FC-LS.pdf" page="154"/>
- Payload 包含：
  - 整个 Fabric 的 Service Parameters
  - 64-bit `F_Port_Name`
  - 64-bit `Fabric_Name`。<cite doc="FC-LS.pdf" page="154"/>

成功后，若新的 `N_Port_ID`、`F_Port_Name`、`Fabric_Name` 与此前 Fabric Login 相同，则 Nx_Port 可继续与其它 Nx_Port 操作；否则可继续进行 N_Port Login。<cite doc="FC-LS.pdf" page="154"/>

### 5.2 对端其实是 Nx_Port：LS_ACC，且 N_Port/F_Port=0

这表示当前链路不是 Fabric，而是**点到点连接到另一个 Nx_Port**：

- `N_Port/F_Port` 位为 `0`；<cite doc="FC-LS.pdf" page="154"/><cite doc="FC-LS.pdf" page="168"/>
- `D_ID` 应为原 FLOGI 的 `S_ID`；<cite doc="FC-LS.pdf" page="154"/>
- Payload 包含：
  - 来自 FLOGI 的 Service Parameters，但**所有 classes 均标为 invalid**
  - 对端的 64-bit `N_Port_Name`
  - 对端的 64-bit `Node_Name`。<cite doc="FC-LS.pdf" page="154"/>

然后根据 `N_Port_Name` 比较决定谁发起后续 PLOGI：

| 比较结果 | 动作 |
|---|---|
| 收到的 `N_Port_Name` 小于本端 `N_Port_Name` | 本端继续进行 N_Port Login（PLOGI）。<cite doc="FC-LS.pdf" page="154"/> |
| 收到的 `N_Port_Name` 大于本端 `N_Port_Name` | 本端等待对端发来 PLOGI。<cite doc="FC-LS.pdf" page="154"/> |

### 5.3 `F_BSY`

若收到 `F_BSY`：

- `OX_ID` 与原 FLOGI 相同；<cite doc="FC-LS.pdf" page="154"/>
- `D_ID` 为原 FLOGI 的 `S_ID`；<cite doc="FC-LS.pdf" page="154"/>
- 表示 Fabric 忙；Nx_Port **可稍后重试 FLOGI**。<cite doc="FC-LS.pdf" page="154"/>

### 5.4 `P_BSY`

若收到 `P_BSY`：

- `OX_ID` 与原 FLOGI 相同；<cite doc="FC-LS.pdf" page="154"/>
- `D_ID` 为原 FLOGI 的 `S_ID`；<cite doc="FC-LS.pdf" page="154"/>
- 表示这是点到点连接，且对端 Nx_Port 当前忙；Nx_Port 可在延迟后继续进行 N_Port Login，以等待对端不再 busy。<cite doc="FC-LS.pdf" page="154"/>

### 5.5 `F_RJT`

若收到 `F_RJT`，表示 Fabric 拒绝 FLOGI。之后动作由 Payload 中的 reason code 决定：

- 若 reason code 是 **“Class not supported”**，Nx_Port 可换一个 class 重新发起 FLOGI。<cite doc="FC-LS.pdf" page="154"/>
- 若 reason code 是 **“Invalid S_ID”**，则 Nx_Port 可换一个 `S_ID` 再发 FLOGI：  
  - 如果被拒的是 `000000h` 或 `0000h || YY`，则 Nx_Port 可通过标准外方法选一个 24 bit 地址 `XXXXXX`，并以其作为新 `S_ID` 重发；<cite doc="FC-LS.pdf" page="154"/>
  - 如果被拒的是 `XXXXXX`，则可改为 `00 00 00`、`00 00 yy`，或另一个新的 `XX XX XX` 再重发。<cite doc="FC-LS.pdf" page="154"/>

### 5.6 `P_RJT`

若收到 `P_RJT`，表示对端其实是另一个 Nx_Port，且对点到点场景拒绝了本次 FLOGI：

- 若 reason code 是 **“Class not supported”**，Nx_Port 可改用不同 class 继续 N_Port Login；<cite doc="FC-LS.pdf" page="154"/>
- 其它 reason code 则按对应原因处理。<cite doc="FC-LS.pdf" page="154"/>

## 6. FLOGI 相关机制：实现时必须一起理解的几个点

## 6.1 显式登录 vs 隐式登录

- 标准允许显式登录和隐式登录。<cite doc="FC-LS.pdf" page="152"/>
- 显式登录就是用 FLOGI/PLOGI ELS 在新 Exchange 中交换 Service Parameters。<cite doc="FC-LS.pdf" page="152"/>
- 隐式 Fabric Login **不要求** Nx_Port 一定支持 FLOGI 请求/应答。<cite doc="FC-LS.pdf" page="152"/>

所以如果用户问“FLOGI 的完整流程”，通常指的是**显式 Fabric Login**流程；而不是所有可能的隐式实现。

## 6.2 登录是长生命周期关系

Login（不管是与 Fabric 还是与其它 Nx_Port）是 long-lived 的，并不与 Class 1 dedicated connection 一一对应。<cite doc="FC-LS.pdf" page="152"/>  
因此 FLOGI 成功后得到的是“运行环境建立”，不是单次报文交易。

## 6.3 Common Features 中几个最关键位

结合表 150，FLOGI 里实现最需要关心的 Common Features 有：

| 位/字段 | 含义 | 对 FLOGI 的意义 |
|---|---|---|
| `N_Port/F_Port` | 0=N_Port，1=F_Port | 用来区分“真 Fabric 成功”还是“其实是点到点 Nx_Port 对端”。<cite doc="FC-LS.pdf" page="168"/><cite doc="FC-LS.pdf" page="154"/> |
| `Virtual Fabrics bit` | 是否参与 VF 协商 | 是后续 EVFP/多 VF 登录的入口。<cite doc="FC-LS.pdf" page="167"/><cite doc="FC-LS.pdf" page="203"/> |
| `Payload Bit` | 是否存在扩展 Payload | 决定 Services Availability / Login Extension Data 等扩展字段是否出现。<cite doc="FC-LS.pdf" page="168"/><cite doc="FC-LS.pdf" page="166"/> |
| `Buffer-to-buffer Credit` | 链路缓冲信用 | FLOGI/FLOGI LS_ACC 中用于链路参数协商。<cite doc="FC-LS.pdf" page="167"/><cite doc="FC-LS.pdf" page="170"/> |

## 6.4 Buffer-to-buffer Credit 约束

标准对该字段给出约束：

- 定义接收用于保存 Class 1/6 connect-request、Class 2、Class 3 帧的缓冲数；<cite doc="FC-LS.pdf" page="170"/>
- 取值范围是 `1..32767`，`0` 保留。<cite doc="FC-LS.pdf" page="170"/>
- 对 N_Port Login 来说，仅在点到点拓扑或同环的两个 NL_Port 间有意义。<cite doc="FC-LS.pdf" page="170"/>

## 7. FLOGI 与 Virtual Fabrics / EVFP 的关系

这是 FC-LS 里 FLOGI 很重要、也最容易漏掉的扩展机制。

## 7.1 VF 协商从 FLOGI 的 Virtual Fabrics bit 开始

Virtual Fabrics 的使用是在执行 FLOGI 时，通过 FLOGI Common Service Parameters 里的 `Virtual Fabrics bit` 来协商的。<cite doc="FC-LS.pdf" page="203"/>

语义是：

- 在 **FLOGI 请求**里该 bit = 1：表示发送方 N_Port **可以协商 Virtual Fabrics 参数**。<cite doc="FC-LS.pdf" page="203"/>
- 在 **FLOGI LS_ACC**里该 bit = 1：表示 Fabric **要求 N_Port 发起 EVFP 事务**。<cite doc="FC-LS.pdf" page="203"/>

表 177给出了四种组合：

| 请求方 VF bit | F_Port 配置 | 返回行为 |
|---|---|---|
| 0b | VF not allowed | `LS_ACC` with VF bit = 0b |
| 1b | VF not allowed | `LS_ACC` with VF bit = 0b |
| 0b | VF allowed | `LS_ACC` with VF bit = 0b |
| 1b | VF allowed | `LS_ACC` with VF bit = 1b |

<cite doc="FC-LS.pdf" page="203"/>

换句话说：**只有“请求方想协商 + F_Port 也允许”时，响应里才回 1b。**<cite doc="FC-LS.pdf" page="203"/>

## 7.2 当 FLOGI LS_ACC 的 VF bit=1 时，会发生什么

若 F_Port 要求后续 EVFP：

- FLOGI `LS_ACC` 的 `D_ID` 必须置为 `000000h`；<cite doc="FC-LS.pdf" page="203"/>
- 这样链路流控参数（如 `Buffer-to-buffer Credit` 和 `Buffer-to-buffer Receive Data Field size`）已经完成协商，但**此时并没有给 N_Port 分配 `N_Port_ID`**。<cite doc="FC-LS.pdf" page="203"/>
- N_Port 之后可用 `N_Port Controller Well Known Address`（`FFFFF0h`）作为 `S_ID` 去做 Authentication 或 EVFP 事务。<cite doc="FC-LS.pdf" page="203"/>

这点很关键：  
**FLOGI 成功返回 LS_ACC，不一定意味着已经拿到可用的 N_Port_ID；如果 VF 协商被要求，可能先只完成链路基础参数协商。**<cite doc="FC-LS.pdf" page="203"/>

## 7.3 带 Virtual Fabrics 的初始化状态机（实现流程视角）

8.2 章给出了实现导向的状态机式流程。抽象成文字如下：

### 阶段 1：基础 FLOGI（P1）

- N_Port 先执行 FLOGI。<cite doc="FC-LS.pdf" page="204"/>
- 在请求中使用与 Port VF_ID 关联的 Logical N_Port 的 `N_Port_Name`；在响应中，F_Port 使用与 Port VF_ID 关联的 Virtual Fabric 的 `Fabric_Name`。<cite doc="FC-LS.pdf" page="204"/>

### 阶段 2：根据 LS_ACC 的 Security / VF bit 决定后续

| 条件 | 转移 |
|---|---|
| Security=1 | P1→P2，先认证。<cite doc="FC-LS.pdf" page="205"/> |
| Security=0 且 VF bit=0 | P1→P3，直接进入正常 N_Port 运行。<cite doc="FC-LS.pdf" page="205"/> |
| Security=0 且 VF bit=1 | P1→P4，进入 EVFP 处理。<cite doc="FC-LS.pdf" page="205"/> |
| 认证成功且 VF bit=0 | P2→P3。<cite doc="FC-LS.pdf" page="205"/> |
| 认证成功且 VF bit=1 | P2→P4。<cite doc="FC-LS.pdf" page="205"/> |

如果在认证路径上，且 FLOGI LS_ACC 的 VF bit=1 且 `D_ID=000000h`，则认证时要用 `N_Port Controller WKA` 作为 `S_ID`；否则用已分配的 Address Identifier。<cite doc="FC-LS.pdf" page="205"/>

### 阶段 3：EVFP 协商（P4）

- N_Port 发起 EVFP 处理。<cite doc="FC-LS.pdf" page="205"/>
- EVFP 协商决定是否启用 VFT tagging、Port VF_ID 是否一致、链路上哪些 VF 生效。<cite doc="FC-LS.pdf" page="206"/>

### 阶段 4：每个 VF 各自做一次带标签的 FLOGI（P5(k)）

一旦 EVFP 结果表明启用 VFT tagging，就会对每个协商出的 Virtual Fabric `K` 进入 `P5(k)`：

- N_Port 发出的 FC 帧带 `VFT_Header`，承载 `VF_ID = K`；<cite doc="FC-LS.pdf" page="205"/>
- 发送一个带 `VF_ID K` 标签的 FLOGI 请求；<cite doc="FC-LS.pdf" page="205"/>
- 该请求携带该逻辑 N_Port 的 `N_Port_Name`，以及 VF K 的操作参数（超时、服务类别等）；<cite doc="FC-LS.pdf" page="205"/>
- 该 FLOGI 的 LS_ACC 为该逻辑 N_Port 在 VF K 内分配一个 `N_Port_ID`。<cite doc="FC-LS.pdf" page="205"/>

但这里有 4 个 Common Service Parameters **被忽略**，而沿用 P1 中的值：

- `Buffer-to-buffer Credit`
- `BB_Credit Management`
- `BB_SC_N`
- `Buffer-to-buffer Receive Data_Field size`。<cite doc="FC-LS.pdf" page="205"/>

### 阶段 5：每个 VF 内可能继续认证，再进入运行

- 若该 VF 内 FLOGI 结束后 `Security bit=1`，则 `P5(k)→P6(k)`，进入该 VF 内认证。<cite doc="FC-LS.pdf" page="205"/>
- 若 `Security bit=0`，则 `P5(k)→P7(k)`，直接进入逻辑 N_Port 运行。<cite doc="FC-LS.pdf" page="206"/>
- 认证成功则 `P6(k)→P7(k)`。<cite doc="FC-LS.pdf" page="206"/>

### 阶段 6：若 EVFP 决定不启用 Tagging

- 若 EVFP 判断不启用 VFT tagging，且 N_Port 与 F_Port 的 `Port VF_ID` 相同，则 `P4→P1`，并要求 N_Port **重新做一次 FLOGI，但请求中的 Virtual Fabrics bit 置 0**，以获取 Port VF_ID 对应 VF 中的 `N_Port_ID`。<cite doc="FC-LS.pdf" page="206"/>
- 若 `Port VF_ID` 不同，则 `P4→P8`，进入 Isolated/offline。<cite doc="FC-LS.pdf" page="206"/>

## 8. 可以把 FLOGI 记成的“实现流程图”

## 8.1 普通（非 VF 扩展）FLOGI

```text
Nx_Port
  └─ 新建 Exchange
      └─ 发送 FLOGI
           - ELS code = 04h
           - D_ID = FFFFFEh
           - S_ID = 000000h / 0000h||YY / XXXXXX
           - Payload = Service Params + N_Port_Name + Node_Name
      └─ 等待响应
           ├─ LS_ACC, N_Port/F_Port=1
           │    ├─ 获得 Fabric 参数 + F_Port_Name + Fabric_Name
           │    ├─ 获得/确认 N_Port_ID
           │    └─ 后续进入正常运行 / PLOGI / 认证 / VF协商
           ├─ LS_ACC, N_Port/F_Port=0
           │    └─ 识别为点到点 Nx_Port，对比 N_Port_Name 后决定谁发 PLOGI
           ├─ F_BSY
           │    └─ 稍后重试
           ├─ P_BSY
           │    └─ 延迟后进行 N_Port Login
           ├─ F_RJT
           │    └─ 按 reason code 改 class 或改 S_ID 重试
           └─ P_RJT
                └─ 按 reason code 处理点到点后续
```

依据：<cite doc="FC-LS.pdf" page="153"/><cite doc="FC-LS.pdf" page="154"/>

## 8.2 含 Virtual Fabrics 的扩展 FLOGI

```text
P1: 初始 FLOGI
  ├─ Security=1        → P2 AUTH_ELS
  ├─ Security=0,VF=0   → P3 正常运行
  └─ Security=0,VF=1   → P4 EVFP

P2: AUTH_ELS
  ├─ 认证成功且VF=0 → P3
  └─ 认证成功且VF=1 → P4

P4: EVFP
  ├─ 启用 VFT tagging → 对每个 VF 进入 P5(k)
  ├─ 不启用 tagging 且 Port VF_ID 相同 → 回 P1，用 VF bit=0 再做一次 FLOGI
  └─ 不启用 tagging 且 Port VF_ID 不同 → P8 Isolated

P5(k): 带 VF_ID=k 标签的 FLOGI
  ├─ Security=1 → P6(k)
  └─ Security=0 → P7(k)

P6(k): VF 内认证
  └─ 成功 → P7(k)

P7(k): Logical N_Port 正常运行
```

依据：<cite doc="FC-LS.pdf" page="204"/><cite doc="FC-LS.pdf" page="205"/><cite doc="FC-LS.pdf" page="206"/>

## 9. 最后给一个“抓重点版”

### 9.1 一句话定义
FLOGI 是 Nx_Port 面向 Fabric 的登录 ELS，用来发现 Fabric、交换基础服务参数、分配/确认 `N_Port_ID`，并为认证和 Virtual Fabrics 协商打开后续流程。<cite doc="FC-LS.pdf" page="153"/>

### 9.2 三个最重要的报文事实
1. FLOGI 的 ELS 命令码是 `04h`。<cite doc="FC-LS.pdf" page="37"/>  
2. 请求在新 Exchange 中发给 `D_ID = FFFFFEh`。<cite doc="FC-LS.pdf" page="153"/>  
3. Payload 核心结构是：`ELS_Command + Common Service Parameters + Port_Name + Node/Fabric_Name + 各 Class Service Parameters + 可选扩展`。<cite doc="FC-LS.pdf" page="166"/>

### 9.3 三个最重要的流程判断点
1. **LS_ACC 的 `N_Port/F_Port` 位**：判断对端是 Fabric 还是点到点 Nx_Port。<cite doc="FC-LS.pdf" page="154"/><cite doc="FC-LS.pdf" page="168"/>  
2. **`S_ID` 的取值**：决定是“让 Fabric 全分配地址”还是“保留低 8 bit”还是“请求确认现有地址”。<cite doc="FC-LS.pdf" page="153"/>  
3. **Virtual Fabrics bit**：决定是否进入 EVFP，再为每个 VF 重新做带标签的 FLOGI。<cite doc="FC-LS.pdf" page="203"/><cite doc="FC-LS.pdf" page="205"/>
