# PageIndex 观察总结（优势 / 回答风格 / 复现要点）

来源：本目录 `log.md`（真实 PageIndex 对 FC-LS.pdf 的问答）+ 早前 FLOGI/Table149 对话。
目的：作为 V7 复现 PageIndex 的参照。当前 V7 已部分复现但效果不好——本文给出"它强在哪、怎么做的、V7 可能差在哪"。

---

## 1. 核心优势（它为什么好）

```
① 结构导航 + 索引跳跃，不盲扫：
   先 get_document_structure() 拿【章节大纲(标题+页码+摘要)】→ 用摘要【推断目标位置】→ 只读那几页。
   例：要 Table 149 → 在结构里看到"6.6.2.1 … Table 150 … start_index:167" → 推 149 在 167 前 → 读 163-166 命中。
   → 210 页文档，4 页范围内精准定位，不逐页扫。

② 完整性 = 全面收集 + 结构化整合（它自己的总结）：
   "长文质量 = 信息收集是否全面(覆盖所有相关页) + 整合是否有结构(不是把原文堆一起)"。
   一个大问题先拆【维度】(流程/帧结构/响应/完成条件/SOF…)，每维度定位到节/页，【一次性读全相关页】。

③ 跨页表：一次请求多页 + Continued 标记拼接：
   Table 150 跨 167-169 → 请求 "167-169" 一次拿连续内容；表头只在首页，后续页标 (Continued)；
   agent 识别标记，把后续行【归到同一张表的表头列定义】下。

④ 复用已读：Table 149/150 读过 → 后面问 FLOGI 直接引用，不重复调用。

⑤ 每块带 <cite doc page>，不凭空生成；诚实标边界（如自环"超出标准范围"）。
```

---

## 2. 工具集 + 工作流（复现的骨架）

PageIndex 自述 **9 个工具**，核心日常是【读文档的 3 个】：

```
库导航：get_folder_structure() / browse_documents() / search_documents()
处理  ：process_document() / get_document()
读文档（核心3个）：
   get_document_structure()  ← 章节大纲(标题+页码+【摘要】)，用于【定位】内容在哪几页
   get_page_content()        ← 读【实际文字 + 表格(转成 Markdown 表格文本)】，支持【一次多页】
   get_document_image()      ← 页面里的图(纯图表格/图需先从 page_content 拿路径再调这个)
管理：remove_document()

标准工作流：
   大文档(>20页)：get_document_structure() → get_page_content()(精准定位页)
   小文档(≤20页)：get_page_content() 直接读
   读图：get_page_content()(拿图路径) → get_document_image()
   "结构工具负责导航，内容工具负责读取"
```

> 对应 V7 已有文件：`structure_chunker.py`→结构、`build_content_db.py`/`page_index.py`→内容索引。复现就是把这套工具+loop 做对。

### get_page_content 返回什么（关键）
PDF 表格 → **Markdown 表格文本**（PDF→md 的转换在【工具里】，不是 LLM 现编），例：
```
| Bits Word | 31 .. 24 | 23 .. 16 | 15 .. 08 | 07 .. 00 |
| 0 | ELS_Command code |
| 1 | MSB | Common Service Parameters (16 bytes) |
```
能做：提取文字/表格 md、脚注、一次多页、嵌入图片【路径】。
不能：直接"看"纯图片表格、保留复杂合并单元格视觉结构、自动关联非连续页（需 agent 判断）、解析图片内容（需 get_document_image）。

## 3. 回答风格（输出特征）

```
· 重度【Markdown 结构化】：表格(枚举/矩阵)、ASCII 流程图/时序图/帧字段树、分层列表、多级标题。
· 组织顺序【宏观→微观】：① 定位/前提 ② 目的(原文列表) ③ 帧结构(最具体) ④ 交互流程(时序+分支表) ⑤ 机制细节 ⑥ 全景图。
· 规范语言 → 结构化：条款式英文 → 表格/流程图/分层列表（不照抄原文）。
· 【加逻辑连接】：原文只说"是什么"，它补"为什么""然后怎样"（如 FLOGI→PLOGI→PRLI 全景）。
· 每块 <cite doc page>；诚实标边界与例外。
```

## 4. "长文输出"三阶段（answer recipe，直接可抄进 agent prompt）

```
阶段1 信息收集（工具）：
   先想【这问题涉及哪些维度】→ 每维度对应到节(流程§6.2 / 帧 Table149 / 响应§6.2.2.3 / 完成§6.2.5-6 / SOF§6.2.3)
   → 用 structure 定位页码 → 【一次性读全相关页】(get_page_content "153,154,155,156") → 已读的复用不重调。
阶段2 信息整合（脑内）：
   原文散落 → 识别维度 → 跨页合并同主题(完成条件散在155/156→合一) → 规范条款转结构化(表/图/列表) → 补逻辑连接。
阶段3 结构化输出：
   宏观→微观分块，每块带 <cite>，不凭空生成。
核心原则：不盲读全文 / 复用已读 / 分层不堆砌 / 原文→结构化 / 加逻辑连接。
本质："工具取数据，LLM 理解+重组；质量 = 收集全面(覆盖所有相关页) + 整合有结构。"
```

## 5. 复现要点 / V7 效果不好的可能原因（诊断清单）

按"完整性 = 收集全面 + 结构化整合"两侧查：

```
A 导航/定位（收集是否全面）：
   □ get_document_structure 有没有【高质量摘要 + 准确页码】？没有 → 无法索引跳跃 → 盲找/漏页。← 最常见的"效果不好"根因
   □ agent 有没有先做【维度拆解 → 定位 → 一次读全相关页】？没有 → 收集不全 → 答不完整。
   □ 有没有【复用已读】？没有 → 重复调、慢、可能不一致。
B 内容读取（表/图质量）：
   □ get_page_content 的【PDF→Markdown 表格】转得干净吗？garbled → 字段/条件丢 → 这是 V7 最该先测的。
   □ 跨页：能【一次多页】+ 识别 (Continued) 拼接吗？不能 → 跨页表丢行。
   □ 纯图片表格有没有走 get_document_image（vision）兜底？没有 → 图里的表读不到。
C 输出（整合是否有结构）：
   □ 输出是【宏观→微观分块 + 表/流程图 + cite】，还是【堆原文】？堆原文 = 看着就"不如 PageIndex"。
   □ 有没有【补逻辑连接】（为什么/然后）？没有 → 干巴、不像 PageIndex。
```

## 6. 对 V7 抽取的启示（这套读法怎么接到 IR）

```
PageIndex 的"读"= V7 架构的【读层】，它产出的【完整结构化 md（带 cite）】= V7 的【完整忠实中间体】。
→ 复现 PageIndex 把"读层 + 完整中间体"做出来；
→ 再在它【之上】加 V7 的贡献：字段级 provenance(把页级 cite 细化到 span/cell) + 结构化成 IR + prior-gate verify + 生成/审计。
注意：PageIndex 的 cite 是【页级】，V7 audit 要更细；PageIndex 止于"读给人看"，V7 要"读→grounded IR→可运行生成"。
红线不变：md 是可核中间体不是事实源；事实源是原文 span / 表 cell / 图 crop。
```
