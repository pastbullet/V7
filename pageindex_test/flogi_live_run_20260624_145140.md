下面按“定位 → 帧/Payload 结构 → 完整流程 → 关键机制/分支 → 实现视角总结”的顺序，系统梳理 FLOGI。所有具体结论都只基于已读取页面。

---

## 1. FLOGI 在 FC-LS 里的定位

FLOGI（F_Port Login）是一个 ELS（Extended Link Service）命令；ELS 请求/应答都在单个 Exchange 内完成，且 ELS 帧的 `TYPE` 固定为 `01h`。<cite doc="FC-LS.pdf" page="36"/>  
在 ELS 通用规则里，如果 Fabric 存在，则 **FLOGI 必须先于其他 ELS**。<cite doc="FC-LS.pdf" page="36"/>  
表 3 进一步给出：`FLOGI` 的 `ELS_Command code` 是 `04h`，描述为 `F_Port Login`。<cite doc="FC-LS.pdf" page="37"/>

从登录章节看，Login 的目标是让 Nx_Port 与 Fabric（若存在）以及后续通信对象建立运行环境；Fabric Login 和 N_Port Login 使用相似过程，但 `D_ID`、可能的 `S_ID` 不同。显式 Login 使用 FLOGI 或 PLOGI 这类 Login ELS，在新的 Exchange 中传送本端 Service Parameters，而 `LS_ACC` 返回对端的 Service Parameters。<cite doc="FC-LS.pdf" page="152"/>

---

## 2. FLOGI 的作用：它到底完成什么

文档把 Fabric Login 的作用列得很清楚：

1. 判断是否存在 Fabric；  
2. 若存在，向 Nx_Port 提供整个 Fabric 的运行特征、`F_Port_Name` 和 `Fabric_Name`；  
3. 若存在，把 Nx_Port 的运行特征、`N_Port_Name` 和 `Node_Name` 提供给 Fabric；  
4. 若存在，Fabric 可为发起方分配或确认 `N_Port_ID`；  
5. 初始化 buffer-to-buffer Credit；  
6. 若双方支持认证，开启后续 Nx_Port-to-Fabric Authentication；  
7. 若双方支持 Virtual Fabrics，开启后续 VF 参数协商。<cite doc="FC-LS.pdf" page="153"/>

同时，**未完成 Fabric Login 前，不得尝试与其他 Nx_Ports 通信**。<cite doc="FC-LS.pdf" page="153"/>

---

## 3. 通用 ELS 帧规则：FLOGI 所依附的外层封装

### 3.1 Exchange / Sequence 规则

- 一个 ELS 请求或应答由 **单个 Sequence** 组成；  
- `ELS_Command code` 位于 **该 Sequence 首帧 Payload 的第一个字**；  
- 一个 ELS 请求及其应答必须在 **同一个 Exchange** 内完成；  
- `LS_ACC` 通过在应答最后一帧把 **Last Sequence bit 置 1** 来结束该 Exchange；  
- 若请求或应答有后续帧，这些后续帧只承载补充 Payload，不重复 `ELS_Command code`。<cite doc="FC-LS.pdf" page="36"/>

### 3.2 TYPE / R_CTL

- `TYPE = 01h`（ELS 帧固定值）。<cite doc="FC-LS.pdf" page="36"/>  
- `R_CTL` 按表 2：  
  - `0010b` = Request  
  - `0011b` = Reply  
  - `0001b` = Solicited Data（仅特殊 CSU ELS 使用）<cite doc="FC-LS.pdf" page="36"/>

## 4. FLOGI/PLOGI/LS_ACC 的总 Payload 版式

表 149 给出了 **FLOGI、PLOGI 或 LS_ACC** 的通用 Login Payload 版式：

| Word 范围 | 内容 |
|---|---|
| 0 | `ELS_Command code` |
| 1–4 | `Common Service Parameters`（16 bytes） |
| 5–6 | `Port_Name` |
| 7–8 | `Node_Name` 或 `Fabric_Name` |
| 9–12 | `Class 1 and Class 6 Service Parameters`（16 bytes） |
| 13–16 | `Class 2 Service Parameters`（16 bytes） |
| 17–20 | `Class 3 Service Parameters`（16 bytes） |
| 21–24 | Obsolete |
| 25–28 | `Vendor Version Level`（16 bytes） |
| 29–30 | `Services Availability`（8 bytes）* |
| 31 | `Login Extension Data Length` * |
| 32–61 | Reserved * |
| 62–63 | `Clock Synchronization QoS`（8 bytes）* |
| 64–n | `Login Extension Data` * |

其中带 `*` 的字段只有在 **Payload Bit=1** 时才出现；若 `Payload Bit=0`，这些字段不存在，此时 Payload 长度为 **116 bytes**。<cite doc="FC-LS.pdf" page="166"/>

这意味着：**FLOGI 本质上是“ELS 外层 + Login 通用载荷 + FLOGI 专用适用性规则”**。<cite doc="FC-LS.pdf" page="166"/>

## 5. FLOGI 请求帧：发起时要怎么填

### 5.1 发送条件与地址填写

显式 Fabric Login 要求 Nx_Port 发送一个 `FLOGI` ELS。<cite doc="FC-LS.pdf" page="153"/>  
该 FLOGI：

- 必须在 **新的 Exchange** 中发送；<cite doc="FC-LS.pdf" page="153"/>
- 由 Nx_Port 分配 `OX_ID`；<cite doc="FC-LS.pdf" page="153"/>
- `D_ID` 必须设为 **众所周知的 F_Port 地址 `FFFFFEh`**。<cite doc="FC-LS.pdf" page="153"/>

### 5.2 Payload 内容

FLOGI Payload 包含：

- 发起方的 `Service Parameters`；
- 64-bit `N_Port_Name`；
- 64-bit `Node_Name`。<cite doc="FC-LS.pdf" page="153"/>

这些 Service Parameters 的总体结构与适用性由 6.6 节及表 150/155/151/156 等决定。<cite doc="FC-LS.pdf" page="153"/><cite doc="FC-LS.pdf" page="167"/><cite doc="FC-LS.pdf" page="169"/><cite doc="FC-LS.pdf" page="179"/>

### 5.3 未标识端口时的 S_ID 语义

若 Nx_Port 尚未标识，它在 FLOGI 里的 `S_ID` 有两种合法请求方式：

| S_ID 填法 | 含义 |
|---|---|
| `000000h` | 请求 Fabric 分配全部 24 位 `N_Port_ID` |
| `0000h || YY` | 请求 Fabric 分配高 16 位（23:8），并校验低 8 位（7:0） |

这里低 8 位就是 `AL_PA`。<cite doc="FC-LS.pdf" page="153"/>

## 6. FLOGI 的 Payload 内部结构：重点字段怎么理解

## 6.1 Common Service Parameters（FLOGI 请求）

文档明确说：FLOGI 的 Common Service Parameters Payload 见 **表 151**。<cite doc="FC-LS.pdf" page="169"/>

表 151 给出 FLOGI 请求中 Common Service Parameters 的布局：

| Word | 内容 |
|---|---|
| 0 | `FC-PH Version - obsolete` + `Buffer-to-buffer Credit` |
| 1 | `Common Features` + `BB_SC_N` + `Buffer-to-buffer Receive Data Field size` |
| 2 | Reserved |
| 3 | Reserved |

但这是“版式表”；哪些字段对 FLOGI 真正有意义，要结合 **表 150（适用性）** 看。<cite doc="FC-LS.pdf" page="169"/><cite doc="FC-LS.pdf" page="167"/><cite doc="FC-LS.pdf" page="168"/>

### 6.1.1 表 150 中与 FLOGI 最相关的 Common 参数

对 FLOGI 请求/LS_ACC 特别关键的点可归纳为：

| 参数 | 位置 | 对 FLOGI 的意义 |
|---|---|---|
| `Buffer-to-buffer Credit` | word 0 bits 15:0 | 对 FLOGI 与 FLOGI LS_ACC 都适用 |
| `Virtual Fabrics bit` | word 1 bit 30 | 对 FLOGI 与 FLOGI LS_ACC 都适用 |
| `N_Port/F_Port` | word 1 bit 28 | 对 FLOGI 与 FLOGI LS_ACC 都适用 |
| `Payload Bit` | word 1 bit 16 | 对 FLOGI 与 FLOGI LS_ACC 都适用；决定后续扩展字段是否存在 |
| `R_T_TOV Value` | word 1 bit 19 所在参数项 | 对 FLOGI 与 FLOGI LS_ACC 都适用 |
| `Clock Synchronization Primitive Capable` | word 1 bit 20 | 对 FLOGI 与 FLOGI LS_ACC 都适用 |
| `Multicast/Broadcast/Hunt Group supported by Fabric` | word 1 bits 25/24/23 | **只对 FLOGI LS_ACC 有意义**，对 FLOGI 请求无意义 |
| `Multiple N_Port_ID Support` | word 1 bit 31（在 FLOGI 列） | 对 FLOGI 请求适用，对 FLOGI LS_ACC 不适用 |
| `Multiple N_Port_ID Assignment` | word 1 bit 29（在 FLOGI LS_ACC 列） | 对 FLOGI LS_ACC 适用，对 FLOGI 请求不适用 |

以上可直接从表 150 的 FLOGI / FLOGI LS_ACC applicability 列读出。<cite doc="FC-LS.pdf" page="167"/><cite doc="FC-LS.pdf" page="168"/>

### 6.1.2 若需要特别关心的数值/约束

- `Buffer-to-buffer Credit` 的取值范围是 **1 到 32767**，`0` 保留。<cite doc="FC-LS.pdf" page="170"/>
- 在表 150 中，`Payload Bit` 默认值是 `0`；它决定表 149 后半部分扩展字段是否出现。<cite doc="FC-LS.pdf" page="168"/><cite doc="FC-LS.pdf" page="166"/>
- `N_Port/F_Port` 的注释说明：`0` 表示 N_Port，`1` 表示 F_Port。<cite doc="FC-LS.pdf" page="167"/>

## 6.2 FLOGI LS_ACC 的 Common Service Parameters

文档说 FLOGI LS_ACC 的 Common Service Parameters Payload 见 **表 153**。<cite doc="FC-LS.pdf" page="170"/>

表 153 的布局为：

| Word | 内容 |
|---|---|
| 0 | `FC-PH Version - obsolete` + `Buffer-to-buffer Credit (Fx_Port)` |
| 1 | `Common Features` + `BB_SC_N` + `Buffer-to-buffer Receive Data Field size` |
| 2 | `R_A_TOV` |
| 3 | `E_D_TOV` |

这和 FLOGI 请求（表 151）形成重要区别：  
**FLOGI 请求的 word 2/3 版式是 Reserved，而 FLOGI LS_ACC 的 word 2/3 承载 `R_A_TOV` 与 `E_D_TOV`。**<cite doc="FC-LS.pdf" page="169"/><cite doc="FC-LS.pdf" page="170"/>

## 6.3 Class Service Parameters（FLOGI 请求 / FLOGI LS_ACC）

文档说：

- FLOGI 的 Payload Class Service Parameters 见 **表 156**；  
- FLOGI LS_ACC 的 Payload Class Service Parameters 见 **表 158**。<cite doc="FC-LS.pdf" page="179"/>

### 6.3.1 表 156：FLOGI 请求中的 Class Service Parameters

| Word | 内容 |
|---|---|
| 0 | `Service Options` + `Initiator Control` |
| 1 | `Recipient Control` + `Reserved` |
| 2 | `Reserved` + `Total Concurrent Sequences` + `Nx_Port End-to-end Credit` |
| 3 | `Reserved` + `Open Sequences per Exchange` + `Reserved` |

<cite doc="FC-LS.pdf" page="179"/>

### 6.3.2 表 158：FLOGI LS_ACC 中的 Class Service Parameters

| Word | 内容 |
|---|---|
| 0 | `Service Options` + `Reserved` |
| 1 | `Recipient Control` + `Reserved` |
| 2 | `Reserved` + `Reserved` |
| 3 | `CR_TOV` |

<cite doc="FC-LS.pdf" page="179"/>

### 6.3.3 Class validity 规则

类服务参数是否有效由 **word 0 bit 31** 的 Class validity bit 决定：

- `0` = Invalid / Class not supported  
- `1` = Valid / Class supported  

若为 `0`，这一整组 16 字节应被忽略；若为 `1`，则该 Class 被支持。<cite doc="FC-LS.pdf" page="179"/>

## 7. FLOGI 的完整流程

下面把 FLOGI 流程分成“发起 → 正常接受 → 非 Fabric/点对点分支 → Busy/Reject/超时分支 → 完成判定”。

### 7.1 发起阶段

1. Nx_Port 决定执行显式 Fabric Login；<cite doc="FC-LS.pdf" page="152"/><cite doc="FC-LS.pdf" page="153"/>
2. 在 **新的 Exchange** 中发送 `FLOGI`；<cite doc="FC-LS.pdf" page="153"/>
3. 填入本端 Service Parameters、`N_Port_Name`、`Node_Name`；<cite doc="FC-LS.pdf" page="153"/>
4. `D_ID = FFFFFEh`；<cite doc="FC-LS.pdf" page="153"/>
5. 若尚无地址，则 `S_ID` 用 `000000h` 或 `0000h||YY` 表达地址分配诉求。<cite doc="FC-LS.pdf" page="153"/>

---

### 7.2 正常成功：收到 Fabric 的 LS_ACC

这是最标准的 FLOGI 成功路径：

- 返回帧是 `LS_ACC`；
- `OX_ID` 与原 FLOGI 相同；
- Common Service `N_Port/F_Port bit = 1`，表示响应方是 `Fx_Port`；
- `LS_ACC` 的 `D_ID` 是 Fabric 分配给本端的 `N_Port_ID`；  
  - 若原 `S_ID = 000000h`，则返回的 `D_ID = XXXXXX`；  
  - 若原 `S_ID = 0000h || YY`，则返回 `D_ID = XXXXYY`；  
  - 若原 `S_ID = XXXXXX`，则返回同值 `XXXXXX`；
- Payload 中包含：
  - 整个 Fabric 的 Service Parameters；
  - 64-bit `F_Port_Name`；
  - 64-bit `Fabric_Name`。<cite doc="FC-LS.pdf" page="154"/>

收到这个成功应答后，Nx_Port：
- 如果新的 `N_Port_ID / F_Port_Name / Fabric_Name` 与之前登录结果一致，可继续已有操作；
- 或继续进入 `N_Port Login`。<cite doc="FC-LS.pdf" page="154"/>

---

### 7.3 非 Fabric 分支：FLOGI 实际打到了另一个 Nx_Port（点对点判定）

如果收到的是：

- `LS_ACC`；
- `OX_ID` 与原 FLOGI 相同；
- Common Service `Nx_Port/F_Port bit = 0`，表示响应方是 `Nx_Port`；

那么这说明当前是 **point-to-point 连接到了另一个 Nx_Port，而不是 Fabric**。此时：

- `LS_ACC.D_ID = FLOGI.S_ID`；
- Payload 带回“来自 FLOGI 的 Service Parameters，但所有 classes 标记为 invalid”，以及该对端的 64-bit `N_Port_Name` 和 64-bit `Node_Name`；  
- 若收到的 `N_Port_Name` **小于** 本端的 `N_Port_Name`，本端继续执行 `N_Port Login`；  
- 若收到的 `N_Port_Name` **大于** 本端的 `N_Port_Name`，本端等待对方发来 `PLOGI`。<cite doc="FC-LS.pdf" page="154"/>

文档在 6.2.2.4 还给出对称规则：  
如果一个 Nx_Port **收到** FLOGI，它也应返回上述这种 `Nx_Port/F_Port bit = 0` 的 `LS_ACC`，并按双方 `N_Port_Name` 的大小关系决定自己是继续 PLOGI 还是等待。<cite doc="FC-LS.pdf" page="155"/>

这一步很关键：  
**FLOGI 不只是“登录 Fabric”，它还承担“探测当前是不是 Fabric、否则识别点对点邻居”的作用。**<cite doc="FC-LS.pdf" page="153"/><cite doc="FC-LS.pdf" page="154"/>

---

### 7.4 Busy / Reject / No Response 分支

#### 7.4.1 `F_BSY`

若收到 `F_BSY`，说明 **Fabric busy**，可以稍后重试 FLOGI。<cite doc="FC-LS.pdf" page="154"/>

#### 7.4.2 `P_BSY`

若收到 `P_BSY`，说明实际上连到一个当前 busy 的 `Nx_Port`；本端可在延迟后继续尝试 `N_Port Login`。<cite doc="FC-LS.pdf" page="154"/>

#### 7.4.3 `F_RJT`

若收到 `F_RJT`，说明 Fabric 拒绝了 FLOGI，请根据 reason code 动作：

- 若 reason = `"Class not supported"`：  
  可换一个 class 重新发起 FLOGI；<cite doc="FC-LS.pdf" page="154"/>
- 若 reason = `"Invalid S_ID"`：  
  - 若被拒请求中的 `S_ID` 是 `000000h` 或 `0000h || YY`，可在标准外方法下自行选择一个 24-bit `N_Port_ID = XXXXXX`，再以此 `S_ID` 重发 FLOGI；  
  - 若被拒请求中的 `S_ID = XXXXXX`，可重新选择 `00 00 00`、`00 00 yy` 或新的 `XX XX XX`，再重发。<cite doc="FC-LS.pdf" page="154"/>

#### 7.4.4 `P_RJT`

若收到 `P_RJT`，说明是点对点的另一个 `Nx_Port` 拒绝。  
- 若 reason = `"Class not supported"`：可用不同 class 继续做 `N_Port Login`；  
- 其他 reason 按相应 code 处理。<cite doc="FC-LS.pdf" page="154"/>

#### 7.4.5 `LS_RJT`

若收到 `LS_RJT`：

- `D_ID` 取 Fabric 分配给本端的 `N_Port_ID`（规则与前述地址分配情况一致）；
- 动作由 Payload 中 reason code 决定；
- 可根据 reason 修改 Service Parameters 后重新发起 FLOGI。<cite doc="FC-LS.pdf" page="155"/>

#### 7.4.6 无响应

若 **No Response**：

- 可能表示物理传输投递错误；
- 应按 FC-FS-2 做 error recovery；
- 恢复后可重新发起 FLOGI。<cite doc="FC-LS.pdf" page="155"/>

#### 7.4.7 自连场景

若收到的 `N_Port_Name` 与自身相等，则视为端口连到自己；该情况超出本标准范围，FLOGI 被丢弃。<cite doc="FC-LS.pdf" page="155"/>

---

### 7.5 完成条件：什么时候算 FLOGI 结束

文档分别定义了发起方与响应方的“Fabric Login completion”。

#### 7.5.1 发起方完成条件

发起 FLOGI 的 Originator 认为 Fabric Login 已结束，当：

- Class 1：已向 `LS_ACC` 发送 ACK（`EOFt` 或 `EOFdt`）；  
- Class 2：已向 `LS_ACC` 发送 ACK（`EOFt`）；  
- Class 3：已收到 `LS_ACC`。<cite doc="FC-LS.pdf" page="156"/>

完成后，buffer-to-buffer Credit 的值被初始化。<cite doc="FC-LS.pdf" page="156"/>

#### 7.5.2 响应方完成条件

响应方认为 Fabric Login 已结束，当：

- Class 1：已收到对 `LS_ACC` 的 ACK（`EOFt` 或 `EOFdt`）；  
- Class 2：已收到对 `LS_ACC` 的 ACK（`EOFt`）；  
- Class 3：已发出 `LS_ACC`。<cite doc="FC-LS.pdf" page="156"/>

成功结束后，同样初始化 buffer-to-buffer Credit。<cite doc="FC-LS.pdf" page="156"/>

---

## 8. FLOGI 之后与后续 PLOGI 的衔接

文档明确写明：

- 若通过 FLOGI 判定 **Fabric 存在**，则后续按 6.3.2.2 执行 `N_Port Login (PLOGI)`；  
- 若通过 FLOGI 判定 **Fabric 不存在**，则按 6.3.2.4 执行点对点 N_Port Login。<cite doc="FC-LS.pdf" page="156"/>

所以从“整体初始化流程”看：

```text
上电/初始化
  → 尝试 FLOGI
      → 若是 Fabric：拿到 Fabric 参数 / 地址 / 名称
          → 再对具体目标做 PLOGI
      → 若其实是点对点 Nx_Port：
          → 用 FLOGI 结果决定谁发 PLOGI
```

这也是为什么 FLOGI 既是“登录”，也是“拓扑与角色探测”的第一步。<cite doc="FC-LS.pdf" page="153"/><cite doc="FC-LS.pdf" page="154"/><cite doc="FC-LS.pdf" page="156"/>

## 9. FLOGI 相关机制：你实现时最该关心的点

## 9.1 显式 vs 隐式 Login

- 标准允许显式和隐式 Login；
- 显式 Login 就是通过 FLOGI/PLOGI ELS 交换 Service Parameters；
- 隐式 Fabric Login **不要求** Nx_Port 必须支持 FLOGI 请求/应答。<cite doc="FC-LS.pdf" page="152"/>

所以若你做的是“显式 FC 登录栈”，FLOGI 是核心；但标准承认还有其他不通过 FLOGI 的实现路径。<cite doc="FC-LS.pdf" page="152"/>

## 9.2 旧参数替换

显式 Fabric Login 会 **替换之前的 Service Parameters**。<cite doc="FC-LS.pdf" page="153"/>

## 9.3 SOF / Class 选择与重试

Fabric Login 只允许在 **Class 1/2/3** 执行；若 Fabric 不支持某一类，FLOGI 可能要换 class 并用相应 SOF 重试。<cite doc="FC-LS.pdf" page="155"/>  
- `SOFc1` 可用于尝试 Fabric Login 或 Relogin；  
- `SOFi1` 不允许；  
- Class 6 不能用于 Fabric Login。<cite doc="FC-LS.pdf" page="155"/>

若收到 `F_RJT`，原因是 `"Class of service not supported by entity at FFFFFEh"`，则应改用另一受支持 SOF 继续尝试，直到完成或所有支持的 SOF 都试尽。若都失败或超时，则 Fabric 与 Nx_Port 不兼容，需要外部干预。<cite doc="FC-LS.pdf" page="155"/><cite doc="FC-LS.pdf" page="156"/>

## 9.4 Relogin 的影响

若之前已经登录过 Fabric：

- 若新的 `N_Port_ID / F_Port_Name / Fabric_Name` 与上次相同，可继续现有通信；  
- 若其中任一发生变化，Nx_Port 必须对所有 Nx_Ports 做 **implicit logout**，并等待一个 `R_A_TOV` 超时后，才能重新发起或接受通信；之后必须重新进行 N_Port Login。<cite doc="FC-LS.pdf" page="155"/>

这说明 FLOGI 不是一次性的“发现过程”，而是影响整个已建立关系集的一致性检查点。<cite doc="FC-LS.pdf" page="155"/>

## 10. 一个面向实现者的“最小可落地流程图”

### 10.1 发起侧

1. 构造 ELS 帧：
   - `TYPE=01h`
   - `R_CTL=Request(0010b)`
   - `ELS_Command=04h(FLOGI)`<cite doc="FC-LS.pdf" page="36"/><cite doc="FC-LS.pdf" page="37"/>

2. 新建 Exchange，分配 `OX_ID`。<cite doc="FC-LS.pdf" page="153"/>

3. 设置地址：
   - `D_ID=FFFFFEh`
   - `S_ID` 按已知/未知地址状态填写。<cite doc="FC-LS.pdf" page="153"/>

4. 填充 Payload：
   - word 0：ELS_Command
   - words 1–4：FLOGI Common Service Parameters
   - words 5–6：N_Port_Name
   - words 7–8：Node_Name
   - words 9–20：Class Service Parameters
   - words 25–28：Vendor Version Level
   - 如 `Payload Bit=1` 再带扩展区。<cite doc="FC-LS.pdf" page="166"/><cite doc="FC-LS.pdf" page="169"/><cite doc="FC-LS.pdf" page="179"/>

5. 发送并等待响应。<cite doc="FC-LS.pdf" page="153"/>

### 10.2 收到响应后的分支

| 收到内容 | 解释 | 后续动作 |
|---|---|---|
| `LS_ACC` + `Nx_Port/F_Port=1` | 真正连到 Fabric | 接受分配/确认的地址，记录 Fabric 参数，继续 PLOGI 或继续已有会话 |
| `LS_ACC` + `Nx_Port/F_Port=0` | 实际是点对点 Nx_Port | 比较双方 `N_Port_Name`，决定自己发 PLOGI 还是等待对方 |
| `F_BSY` | Fabric 忙 | 稍后重试 |
| `P_BSY` | 对端 Nx_Port 忙 | 延迟后继续 N_Port Login |
| `F_RJT` | Fabric 拒绝 | 看 reason，改 class 或改 S_ID 后重试 |
| `P_RJT` | 点对点对端拒绝 | 看 reason，按 PLOGI 规则处理 |
| `LS_RJT` | 语义拒绝 | 按 reason 调整 Service Parameters 后重试 |
| No Response | 传输/投递错误 | 先做 error recovery，再重试 |

## 11. 最后给一个一句话总结

**FLOGI 是 FC 初始化里“先和 Fabric（或伪装成 Fabric 的对端）打招呼”的第一步**：  
它用标准 ELS 外层承载一套 Login Payload，完成 **Fabric 存在性判定、地址分配/确认、Service Parameters 交换、Fabric 名称信息获取、信用初始化，以及后续认证/VF 协商的入口建立**；若发现其实不是 Fabric，而是点对点另一端口，它又自然转入后续 PLOGI 决策流程。<cite doc="FC-LS.pdf" page="153"/><cite doc="FC-LS.pdf" page="154"/><cite doc="FC-LS.pdf" page="156"/>
